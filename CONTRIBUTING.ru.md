# Участие в разработке

Английская версия: `CONTRIBUTING.md`.

Перед изменениями в репозитории прочитайте `AGENTS.md`. Для изменений поведения или архитектуры профайлера также проверьте `_DevelopmentDocs/decisions/` и релевантные заметки в `_DevelopmentDocs/backlog/`.

## Pull Requests

Все изменения в `main` должны проходить через pull request. Держите PR сфокусированным на одном изменении поведения, документации или обслуживания репозитория.

Используйте чеклист из шаблона PR и указывайте локальную команду проверки, которую запускали. Если изменение влияет на поведение runtime-профайлера, обновляйте тесты и пользовательскую документацию в том же PR.

## Именование веток

Используйте имена веток в нижнем регистре ASCII в таком формате:

```text
<type>/<short-kebab-topic>
```

Разрешенные типы:

```text
feature
fix
docs
l10n
test
refactor
chore
security
release
```

Примеры:

```text
feature/render-graph-analytics
fix/overdraw-readback-session
docs/readme-screenshots
l10n/russian-docs
release/2026-6-5-1
```

Правила:

- Используйте ASCII-буквы нижнего регистра, цифры и дефисы в topic.
- Держите ровно один slash между type и topic.
- Используйте `l10n` для изменений документации только по переводу или локализации.
- Не добавляйте секреты, имена хостов, приватные пути, имена клиентов, серийные номера устройств или токены в имена веток.
- Не используйте `main`, `master`, `release` или имена, похожие на теги, например `v1.2.3`, для PR-веток.

## Локальные проверки

Публичные рекомендации для проверок контрибьюторов: [Проверки изменений контрибьюторов](./docs/ru/contributor-checks.md).

Для изменений только документации или метаданных минимум:

```bash
git diff --check
```

Синхронизируйте затронутые локализованные пользовательские docs, если изменение влияет на несколько аудиторий.
