# Docs Layout Decision

## Решение

- `docs/en` и `docs/ru` - документация для конечных пользователей.
- `Assets/Scripts/SGG.PerfMeter/.Documentation` - package-local пользовательская документация для UPM/package consumers.
- `_DevelopmentDocs` - только внутренние документы разработки.
- `_DevelopmentDocs/backlog` - будущие задачи и полезные, но нереализованные идеи.
- `_DevelopmentDocs/decisions` - короткие объяснения текущих технических решений.
- `_DevelopmentDocs/release` - минимальные внутренние release-prep материалы.

## Что не хранить в `_DevelopmentDocs`

- Публичные продуктовые тексты.
- Историю завершенных итераций.
- Пользовательские guides.
- Длинную теорию, если она не является actionable backlog.

## Почему так

Публичные docs должны быть чистыми и полезными пользователю. Dev docs должны помогать принимать текущие инженерные решения и планировать будущую работу, а не объяснять историю происхождения функций.
